# EP-00002 ‚Äî play-music: random MP3 player from a folder

> **Locked decisions (2026-05-05):** name is `play-music`; default behavior is continuous loop with reshuffle; v1 supports MP3 only.

## Problem / Pain Points

- We want a no-frills CLI music player: point it at a folder of MP3s, hit enter, and music plays in random order. No web UI, no GUI, no library management ‚Äî just shuffle and play.
- Pairs naturally with `yt-mp3`: download a bunch of audio with `yt-mp3` into `output/yt-mp3/`, then `play-music output/yt-mp3` to listen.
- Constraints from `docs/SPECS.md` that shape this tool:
  - One subcommand on the existing Typer entrypoint, package under `src/simple_tools/tools/<name>/`.
  - Stdout discipline: this tool plays audio, not produce file artifacts. Stdout stays mostly silent (or prints "now playing: <path>" lines); errors and progress go to stderr.
  - Per-tool naming: kebab-case command (`play-music`), snake_case package (`play_music`).
  - No new shared "common" module across tools. Stays self-contained.
  - No artifacts to `output/<tool>/` since playback produces nothing on disk; we'll skip the `--output` flag entirely (SPECS already permits this for non-artifact tools).
  - No network calls in tests.

## Suggested Solution

### Tool name

`play-music` ‚Äî locked.

### Playback backend

**Choice: `ffplay` via subprocess.** `ffplay` ships with `ffmpeg`, which we already require for `yt-mp3`. No new Python dependency, no extra system install for users who already followed the `yt-mp3` setup. Headless, no window pop-up with `-nodisp -autoexit -loglevel quiet`.

Alternatives considered and rejected:
- `pygame.mixer` ‚Äî pure-Python control, but adds a heavy dependency (SDL2, pygame itself ~10 MB) for a tool whose job is "spawn a subprocess to play audio."
- `playsound` ‚Äî historically buggy on macOS with newer Python; unmaintained.
- `python-vlc` ‚Äî requires VLC.app installed; redundant when we already need ffmpeg.
- `afplay` (macOS) ‚Äî zero deps but mac-only; would force a per-OS code path.

If ffplay isn't on `PATH`, fail fast with a clear stderr message pointing the user at `brew install ffmpeg`.

### CLI shape

```
simple-tools play-music <folder> [--once] [--no-recursive] [--debug]
```

| Flag | Default | Notes |
|---|---|---|
| `<folder>` (positional) | required | Directory to scan for `.mp3` files. |
| `--once` | off | Play the shuffled playlist once and exit. Default is continuous: when the playlist is exhausted, reshuffle and keep playing. |
| `--no-recursive` | off | Only scan the top level of the folder. Default is to recurse into subfolders. |
| `--debug` | off | Re-raise exceptions instead of one-line stderr error. |

Behavior:
- On startup, scan the folder for `*.mp3` (case-insensitive). If zero found, exit `1` with a clear stderr message.
- Shuffle the list with `random.shuffle`. Spawn `ffplay -nodisp -autoexit -loglevel quiet <path>` per track, blocking on each. Print `‚ñ∂ <path>` to stdout per track for visibility.
- `Ctrl-C` (SIGINT) ends cleanly: terminate the active ffplay subprocess, exit `0`. Don't dump a traceback unless `--debug`.
- `--once` exits 0 after the first pass. Default mode loops, reshuffling between passes.

### Tool layout

```
src/simple_tools/tools/play_music/
‚îú‚îÄ‚îÄ __init__.py       # re-exports the Typer command
‚îú‚îÄ‚îÄ cli.py            # CLI surface (flags, signal handling, exit codes)
‚îî‚îÄ‚îÄ player.py         # discovery + ffplay subprocess wrapper (testable unit)
```

`player.py` exposes:
- `find_mp3s(folder: Path, recursive: bool) -> list[Path]`
- `play_one(path: Path) -> int` (returns ffplay's exit code; raises `FileNotFoundError` if `ffplay` is missing)

`cli.py` orchestrates: discover ‚Üí shuffle ‚Üí loop ‚Üí handle SIGINT ‚Üí exit.

### Tests

`tests/tools/play_music/`:
- `test_player.py` ‚Äî unit tests for `find_mp3s` (recursive on/off, case-insensitive extension, ignores non-mp3) using a `tmp_path` tree of fake files. Mock `subprocess.run` for `play_one`; assert the ffplay command line and how `FileNotFoundError` surfaces.
- `test_cli.py` ‚Äî `CliRunner` with `play_one` patched to return immediately. Verify:
  - Empty folder ‚Üí exit `1`, stderr message
  - Missing folder ‚Üí exit `1`
  - Non-existent ffplay (mocked `FileNotFoundError`) ‚Üí exit `1` with install hint
  - `--once` plays each discovered track exactly once and exits `0`
  - Default mode reshuffles after exhausting the list (test by limiting to 2 passes via a counter on the patch)
  - SIGINT handling: simulated by raising `KeyboardInterrupt` from `play_one`; expect exit `0`, no traceback unless `--debug`

### Resolved decisions

- **Tool name:** `play-music`.
- **Default behavior:** continuous loop with reshuffle between passes; `--once` flag to opt into one-pass-and-exit.
- **Audio formats:** MP3 only for v1. `.m4a` / `.flac` / `.ogg` deferred to a future EP.
